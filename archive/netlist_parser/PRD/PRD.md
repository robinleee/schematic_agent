# 统一硬件 AI 专家系统（审查与诊断）产品需求文档 (PRD) - V3.0 最终交付版

## 1. 产品愿景与定位
打造一个基于 EDA 异构数据图谱与 GraphRAG 技术驱动的 Agentic 硬件辅助系统。通过构建物理单板的“数字孪生”底座并外挂“原厂专家知识库”，实现硬件全生命周期的三大核心闭环：
* **左移防御（原理图审查）：** 在设计阶段，结合 Design Guide 规范自动化巡检全板拓扑，拦截器件选型与电气连接设计违规。
* **右移排障（硬件故障诊断）：** 在调试/售后阶段，接入测试日志，通过上下文感知多维图搜索，毫秒级定位物理硬件的根因故障。
* **终身学习（自主知识获取）：** 具备分级路由安全检索机制与专家反馈闭环，自动解决冷门器件冷启动问题，持续丰富本地硬件知识库。

## 2. 系统整体架构

| 模块名称 | 技术栈与实现方案 | 核心职责 |
| :--- | :--- | :--- |
| **数据底座层 (Graph DB)** | Python ETL 引擎 + Neo4j | 解析 EDA 网表与标准化 BOM，完成属性特征融合，建立包含电气拓扑与物理参数的有向属性图谱。 |
| **知识外脑层 (Vector DB)** | Milvus / Chroma + 嵌入模型 | 存储 Datasheet 和 Design Guide 的切片向量数据，提供非结构化硬件规范的语义检索 (RAG)。 |
| **检索路由层 (Search Router)**| Python 封装的受控 API 接口 | 提供安全的内网 PLM 穿透与外网 API（如 Octopart / 厂商官网）分级定向搜索能力。 |
| **Agent 核心层 (Orchestration)**| **状态机驱动的多智能体框架 (如 LangGraph / OpenClaw)** | **维护持久化多轮对话状态 (State) 避免查询死循环**；提供标准化 Tool Calling 接口（图谱查询、规范检索、联网搜索），调度执行推理任务。 |
| **推理决策层 (LLM Engine)** | 本地部署高性能 LLM (vLLM/AWQ) | 负责意图理解、逻辑拆解、ReAct 循环执行，以及输出人类可读的结构化诊断/审查报告。 |

## 3. 核心功能模块需求

### 3.1 模块一：自动化数据孪生图谱构建 (Data Graph ETL)
* **异构数据融合提取：** 支持导入 Cadence 网表 (`pstxnet.dat` 等) 及 BOM。精准提取拓扑三元组及被动器件的容值、阻值、耐压等物理属性。
* **全局电源树自动化生成：** 自动识别 LDO、Buck 等电源芯片，基于引脚属性正向推导建立全局 `[:POWERED_BY]` 关系网。

### 3.2 模块二：AI 原理图自动化审查 (Schematic Review)
* **动态规则驱动的规范校验：** Agent 自主查询本地知识库中的 Design Guide，获取接口规范（如“I2C 需配置 2.2k 上拉电阻”）。随后调用图谱工具扫描全板 I2C 链路，对违规阻值或缺失电阻的设计进行精准拦截。
* **绝对最大额定值 (AMR) 审查：** 结合图谱中的网络电压属性与 Datasheet 降额规范，自动检索全板滤波电容的耐压值，对存在击穿风险的低耐压选型发出预警。
* **引脚悬空与网络错连审查：** 利用图谱单跳校验能力，结合 Datasheet 提取的 Pin Mux 表格，检查 Open Drain 引脚是否漏接上拉，以及 GND/Power 引脚是否正确连接。

### 3.3 模块三：硬件智能故障诊断 (Hardware Fault Diagnosis)
* **复杂系统 Boot 失败溯源：** 针对开机黑屏等复杂死机故障，Agent 结合芯片上电时序手册与图谱电源树，拆解分析时序违规或关键复位信号链路问题。
* **信号链路中断特异性溯源：** 采用上下文感知的多维路由策略。诊断通信中断时，自动屏蔽 GND 与 VCC 网络，精准提取途经的串联（耦合电容）与并联（ESD 保护）器件并校验其电气参数。
* **多节点共因失效定位：** 当日志显示多传感器同时掉线时，自动切换至图谱电源维度，沿 `POWERED_BY` 边逆向寻址公共祖先电源节点，锁定烧毁的降压芯片或短路电源轨。

### 3.4 模块四：多模态知识外脑与分级检索 (Tiered Knowledge Routing)
为解决冷门芯片的知识盲区并保障核心商业机密，系统设计三级降级检索路由：
* **Tier 1 (高频本地库)：** 优先查询私有 Vector DB 与已固化在 Neo4j 中的常规器件参数，保障毫秒级安全响应。
* **Tier 2 (内网 PLM 穿透)：** 若 Tier 1 缺失，Agent 调用内部 API 查询公司 PDM/PLM 系统（如 Windchill），提取经过内部认证的规格书。
* **Tier 3 (脱敏外网检索)：** 若彻底无记录，触发受限公网搜索。强制执行安全脱敏：剥离所有电路图上下文与公司项目信息，仅允许 Agent 携带干净的制造商编号 (MPN) 访问官方域名或元器件库 API。获取参数后自动沉淀至 Tier 1，实现一次查询，永久本地化。

### 3.5 模块五：对话式协作交互 UI (Conversational Interface)
* **透明化多轮推理展示：** 界面需清晰展示 Agent 的思考链路 (Thought -> Action -> Observation)。
* **可视化图谱联动：** 在输出文本结论的同时，前端联动展示 Neo4j 的局部节点可视化关系网络，让硬件工程师直观确认故障实体。
* **Human-in-the-Loop 专家反馈闭环：** 提供白名单与误报 (False Positives) 标记机制。工程师对报错点击【忽略并加入白名单】后，系统需自动将该规则反向写回 Neo4j 或 Vector DB，实现审查规则的终身自我迭代。

## 4. 非功能性需求 (NFR)
1. **数据红线与隐私隔离：** 系统的 Agent 引擎与大语言模型必须支持纯本地物理机部署。绝不允许将未脱敏的 BOM 描述、网表拓扑或项目代号上传至公有云大模型 API 或通用搜索引擎。
2. **极速图谱响应：** 底层 Cypher 查询（如千节点级别的 `shortestPath` 和模式匹配）的响应延迟需严格控制在 500ms 以内，保障 Agent Tool Calling 的流畅度。
3. **标准化的 BOM 数据准入与拦截机制：** 提供标准化的 CSV BOM 导入模板。解析器在具备合理模糊匹配能力的同时，对严重缺失关键字段（如 Value、Type）的物料，必须在 ETL 阶段抛出异常列表予以拦截，严格拒绝脏数据污染 Neo4j 图谱底座。
4. **大模型上下文防爆与数据截断机制：** 针对 LLM 容易发生的上下文溢出 (OOM) 或注意力丢失问题，所有的图谱 Tool Calling 返回结果必须执行长度校验。当单一查询返回的节点/关系数超过安全阈值（如 50 个）时，系统需强制截断并仅返回统计摘要，同时通过 Prompt 提示 Agent 细化 Cypher 查询条件。

## 5. 落地规划与里程碑建议
* **第一阶段：数字孪生底座建设（图谱 Ready）**
    完成 Cadence 异构网表 (`pstxnet.dat`, `pstchip.dat` 等) 与标准化 BOM 的联合解析脚本，打通 Neo4j 数据库的批量注入与可视化验证。
* **第二阶段：RAG 知识库与检索路由搭建（外脑 Ready）**
    部署本地 Vector DB，跑通基础 Datasheet 导入；开发 Tier 1 到 Tier 3 的分级检索工具，重点完成脱敏 MPN 联网抓取逻辑的测试。
* **第三阶段：Agent 核心闭环编排（系统成型）**
    配置本地大模型推理引擎（如 vLLM），基于 LangGraph 等状态机框架定义系统 Prompt 与工具函数箱。先硬编码跑通“I2C 上拉电阻违规审查”和“PCIe 耦合电容错误排障”两个 MVP 场景。
